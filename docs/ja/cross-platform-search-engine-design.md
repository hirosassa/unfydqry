# クロスプラットフォーム検索エンジン 設計方針

> 🌐 English version: [`../cross-platform-search-engine-design.md`](../cross-platform-search-engine-design.md)

iOS と Android の両方から利用できる共通の全文検索エンジンを **Rust + UniFFI** で実装
するにあたっての設計方針をまとめる。本書は具体的な実装ではなく、設計上の判断とその
根拠を扱う。API やビルド手順はプラットフォーム別ガイド
([`ios.md`](../ios.md) / [`android.md`](../android.md) /
[`flutter-plugin.md`](../flutter-plugin.md))を参照。

本エンジンの設計は、検索の挙動を二つの独立した関心事に分離することを基本とする。

1. 二つの文字列を同一とみなす基準 — 正規化(normalization)
2. クエリと本文を突き合わせる方法 — 検索戦略(search strategy)

いずれもホスト側のバインディングが選択し、実装はすべて Rust の単一コアに置く。
選択肢の組み合わせは要件に応じて変更できる一方、選択された挙動が iOS と Android で
相違することは構造的に生じない。

---

## 1. 全体アーキテクチャ

### 1.1 基本構成 — インデックス所有型

検索エンジンは独自の検索インデックスを所有する。アプリ本体のデータを正
(source of truth)とし、エンジンはそれとは独立した検索インデックスとして存在する。

本体データの保存先はエンジンの関知するところではない。本書で挙げる SwiftData(iOS)や
Room(Android)は代表的な例にすぎず、実際の保存先はリレーショナルDB・ドキュメントストア・
ファイル・リモートサーバなど任意である。エンジンが本体データに要求するのは、各レコードを
安定した ID で再取得できることのみであり、保存形態や保存場所には依存しない。

```
┌─────────────────────────────┐     ┌──────────────────────────────┐
│  iOS アプリ                  │     │  Android アプリ               │
│  ┌────────────────────────┐ │     │ ┌──────────────────────────┐ │
│  │ 本体ストア(正)          │ │     │ │ 本体ストア(正)            │ │
│  └───────────┬────────────┘ │     │ └────────────┬─────────────┘ │
│              │ index/remove  │     │              │ index/remove  │
│  ┌───────────▼────────────┐ │     │ ┌────────────▼─────────────┐ │
│  │ SearchEngine (Swift binding) │ │ │ SearchEngine (Kotlin binding)│
│  └───────────┬────────────┘ │     │ └────────────┬─────────────┘ │
└──────────────┼──────────────┘     └──────────────┼───────────────┘
               │                                    │
        ┌──────▼────────────────────────────────────▼──────┐
        │      Rust コア (UniFFI)  ※実装は物理的に1つ        │
        │   正規化 / インデックス管理 / ランキング / 突き合わせ │
        └───────────────────────────────────────────────────┘
        検索インデックス (本体ストアとは別ファイル)
```

検索インデックスを本体ストアに同居させない理由は二点ある。第一に、本体ストアへ検索用の
列やテーブルを追加すると、そのストアが行うスキーマ検証やクラウド同期(SwiftData や Room
などが典型)と干渉し、正のデータの整合性を損なう可能性がある。第二に、検索インデックスは
正のデータから再構成可能な派生データであり、可用性および整合性の要件が本体ストアと
異なる。このため両者を別ファイルとして分離する。

### 1.2 設計上の重要決定

| 決定事項 | 内容 | 理由 |
|---|---|---|
| 実装の単一化 | 検索ロジックを Rust に1実装だけ置き、各言語へ自動バインディング | アルゴリズムの一致を運用規律ではなく**構造的に保証**する |
| インデックス独立 | 検索インデックスを本体ストアと別に持つ | 本体ストアの整合性チェック・クラウド連携と干渉しない。検索は再構成可能な派生データとして扱う |
| ID のみ返却 | 検索結果は安定キー(ID)とスコアのみを返し、本体は各OSで再フェッチ | エンジンが本体ストアの実装に結合せず、移植性を保つ |
| 挙動はバインディングが選択 | 正規化と検索戦略の**組み合わせ**をホスト側で指定する | 用途ごとに挙動を選択しつつ、実装はコアに一元化する |
| 既定は正規化による畳み込み | 曖昧性は原則として正規化による軸の畳み込みで実現し、近似マッチは選択可能な戦略として併設する | 単純な一致が曖昧検索として機能し、高速かつ決定的。タイプミスの許容など別種の曖昧性は戦略として明示的に選択する |
| ランタイムの同梱 | 正規化処理および検索基盤の実体をOSに依存させず、コアに含める | 端末・OS のバージョン差による結果の差異を排除する(後述の構造的一致保証) |

### 1.3 データフロー

- **書き込み**: 本体ストアへ保存後、正規化前の生テキストを `index(id, text)` でエンジンに渡す(正規化はエンジン内部で実行)
- **検索**: `search(query, limit)` に生入力を渡す。エンジンは同一の正規化を適用してインデックスを参照し、`(id, score)` を返す。本体ストアからは ID で再フェッチする
- **削除**: 本体から削除後、`remove(id)` を呼ぶ

ホストは常に正規化前の文字列のみを渡す。正規化処理をホスト側に露出させないことが、
後述するクロスプラットフォーム一致の前提条件となる。

## 2. 正規化に関する設計方針

正規化は、表記の異なる文字列を共通のキーへ写像する処理であり、検索における曖昧性の
大部分を規定する。本エンジンの正規化方針は、単一の固定された処理から、畳み込む軸を
選択できる合成的なモデルへと変更されている。

### 2.1 正規化を独立した軸の合成として捉える

曖昧性には複数の独立した軸が存在する。大文字小文字、全角半角、かな種別、長音、
繰り返し記号、ダッシュ類、桁区切り、空白の揺れなどである。従来の設計はこれらを単一の
固定処理に束ねていたが、現在は各軸を独立した畳み込みステップとして扱う。

軸を分離する根拠は、要求される曖昧性の範囲が用途によって異なる点にある。全角半角の
統一のみで十分な用途もあれば、かな種別や長音まで吸収すべき用途もある。各軸を独立に
扱えば要件に応じて曖昧性の強度を選択できるが、単一処理に束ねた場合は有効・無効の二択
のみとなる。

### 2.2 基底処理と追加軸の二層構造

正規化は、常に適用される基底処理と、任意に追加される軸の二層から構成される。

- **基底処理(常時適用)**: 全角半角および互換文字の表記差を正準形へ統一する基礎的な
  Unicode 正規化。これは曖昧化ではなく表記の一意化に相当し、すべての構成で適用される。
- **追加軸(選択)**: 小文字化、かな種別の統一、長音の畳み込み、繰り返し記号の展開、
  ダッシュ類の統一、桁区切りの除去、空白の圧縮、Latin 系アクセント記号の除去など。
  ホストが要件に応じて有効・無効を指定する。

選択された軸は固定された正準順序で適用される。順序を固定する目的は、同一の軸集合が
常に同一の結果を生成すること(決定性)を保証し、iOS と Android で同一のキーが得られる
ことを保証する点にある。これにより、挙動は選択された軸の集合のみに依存し、指定順序には
依存しない。

### 2.3 名前付きプリセットの位置づけ

頻出する軸の組み合わせには名称を付与しているが、これは合成モデル上の特定の構成を指す
別名であり、独立した機構ではない。既定のプリセットは大文字小文字・全角半角・かな種別を
畳み込む構成であり、もう一方はかな種別の軸を除いた構成である。プリセットを用いず個別の
軸を指定することも可能である。

既定のプリセットによる畳み込みの例を示す。

```
ガ / が / ｶﾞ      → が     (全半角・かな種別は畳むが、濁点は区別)
カ / か / ｶ       → か     (濁点ありとは別キー)
パ / ぱ / ﾊﾟ      → ぱ
Ｐ / P / ｐ / p   → p
ヴ / ｳﾞ           → ゔ
```

### 2.4 濁点・半濁点の区別

本エンジンは既定で濁点・半濁点を区別する(`か` と `が` を別物として扱う)。日本語検索に
おいて「が」と「か」を同一視することは過剰な曖昧化となる、という判断に基づく。

この区別は、基底処理に採用する正規化形の選択から導かれる。全角半角の統一には互換等価の
正規化が必要だが、その際に合成形(結合済みの単一文字)を採用することで濁点は単一文字
として保持され、区別が維持される。分解形を採用した場合、濁点は独立した結合文字に分離し、
区別を維持する目的に反してキーが不安定化する。詳細は付録A・Bを参照。

濁点の区別は固定的な規則ではなく、選択された軸集合の帰結である。既定の軸集合には濁点を
畳み込むステップが含まれないため、結果として区別される。なお Latin 系のアクセント記号
(café → cafe 等)を畳み込む軸は別に用意されており、日本語の濁点とは独立に制御できる。

### 2.5 正規化とインデックスの対応関係

正規化が可変となったことに伴い、インデックスはその生成に用いた正規化と不可分である、
という制約が生じる。

index 時と search 時に異なる正規化を適用すると、同一の文字列が異なるキーへ写像され、
検索結果が誤る。この種の誤りは検出が困難である。このためエンジンは、インデックス生成に
用いた正規化の識別子を保持し、異なる正規化でインデックスを開いた場合に不整合を明示的に
通知する。これは、誤った結果を返すよりも不整合の時点で処理を停止する方が運用上の安全性が
高い、という判断に基づく。

## 3. 検索戦略に関する設計方針

正規化が二つの文字列を同一とみなす基準を規定するのに対し、検索戦略は正規化済みの
クエリと本文を突き合わせる方法を規定する。本エンジンはこの突き合わせ方法も差し替え
可能とする。

### 3.1 既定戦略 — 辞書非依存かつ順位付け可能な全文検索

既定の戦略は、文字を3文字単位に分割する手法でインデックスを構成し、語の頻度と文書長を
加味した関連度指標で並べ替える。この選択の根拠は二点ある。

- **辞書非依存であること**。形態素解析等の辞書依存の分割は精度が高い一方、辞書の内容が
  環境ごとに異なれば結果に差異が生じる。3文字分割は辞書を用いないため、分かち書きしない
  日本語でも部分一致が成立し、プラットフォーム非依存性を保つ。
- **順位付けが可能であること**。一致集合だけでなく関連度順の結果を返せるため、汎用の
  全文検索として利用できる。

### 3.2 用途に応じた戦略の選択

汎用の順位付け検索が常に最適とは限らない。前方一致はサジェスト・オートコンプリートに、
後方一致は拡張子や敬称の検索に適する。語順を問わない複数語検索、および編集距離や文字
集合の類似度に基づく近似検索も、それぞれ異なる用途で要求される。

本エンジンはこれらをすべてコアに実装し、選択のみをバインディングに委ねる。正規化と
同様に、選択肢はホストへ開放する一方、実装はコアに集約し、いずれの選択でも両OSで
一致する。

近似マッチ(タイプミスの許容)は、正規化では吸収しない別種の曖昧性として、戦略側の関心事に
分離している。正規化は決定的な畳み込みを担い、距離に基づく曖昧性は戦略として選択する、
という役割分担である。

### 3.3 短いクエリの扱い

3文字単位のインデックスは、原理上3文字未満のクエリを参照できない。一方、日本語では
1〜2文字のクエリが頻繁に発生する。このため、3文字未満のクエリは部分一致による検索へ
自動的に切り替える。インデックス方式の制約を利用者に転嫁しないための措置である。

## 4. インデックスのライフサイクルと再生成

### 4.1 生テキストの保持による再生成

正規化方針は将来変更されうる(軸の追加、既定構成の変更など)。その際にホストへ全文書の
再投入を要求する設計は望ましくない。

このためエンジンは、各文書の正規化前テキストを正規化済みの形とともに保持する。これに
より、正規化方針が変更された場合でも、エンジン単独で全文書を再正規化し、インデックスを
再構成できる。ホスト側の操作は不要である。

これは、正規化済みテキストのみを保持し再構築時に再投入を要する当初の方針からの変更で
ある。再生成は明示的に指示できるほか、インデックスを開いた時点で正規化の不整合を検知して
自動的に実行することもできる。

### 4.2 本体ストアとの同期

検索インデックスは本体ストアの派生データであるため、本体の変更をインデックスへ反映する
機構を要する。具体的な機構は採用する本体ストアに依存するが、いずれの場合も変更を捕捉して
該当レコードを再 index する点は共通である。以下は代表的なストアにおける例である。

- **iOS(SwiftData の例)**: 永続的履歴トラッキングにより、前回以降の挿入・更新・削除の
  差分を取得して反映する。削除済みレコードの情報も取得できる設定とすることで、`remove(id)`
  との連携が容易になる。
- **Android(Room の例)**: 変更監視機構、または変更ログを蓄積する方式で変更を捕捉する。
- いずれのストアでも、オフライン編集を伴う場合は本体側に未同期フラグを保持し遅延同期する。

エンジンに与える ID の選択にも設計上の考慮がある。本体ストアの内部識別子
(`PersistentIdentifier` や rowid)に直接結合させず、アプリが付与する安定キー(UUID 等)を
ID に用いる。これによりインデックスが本体ストアの実装から独立し、移植性が保たれる。

## 5. クロスプラットフォーム一致の構造的保証

本設計は、iOS と Android の挙動の一致を、運用上の規律ではなく構造によって保証する。
すなわち、両者が一致しない実装が生じえない構成を採る。

1. **OS 内蔵の言語処理に依存しない**。各OSが備える文字変換やトークナイズは、内部で
   用いる国際化ライブラリのバージョンがOSごとに異なり、結果に差異が生じうる。本エンジンは
   正規化処理の実体をコアに含めるため、結果がビルド時点で確定する。
2. **検索基盤のランタイムを同梱する**。OS 同梱の検索基盤はバージョンが端末ごとに異なり、
   利用可能な機能にも差がある。同一バージョンをコアに同梱し、挙動を固定する。
3. **辞書非依存の処理で土台を構成する**。正規化および既定のインデックス化は辞書を用いない
   決定的処理である。辞書依存の機能(後述の読み検索など)を導入する場合は、辞書をコアに
   同梱してバージョンを固定し、OS の機能は使用しない。
4. **共有の振る舞い仕様(spec)で検証する**。入力に対する期待される正規化結果および期待
   ヒット ID を言語非依存の共有ファイルに記述し、全OSの CI が同一ファイルを検証する。
   コアに差異が生じた場合、すべてのプラットフォームが同一のケースで同時に失敗するため、
   一方のみで差異が生じる事態を検出できる。

これらの基盤となる最上位の原則は、実装を単一に保つことである。バインディングが自動生成
されるため、Swift と Kotlin が異なる実装になることはなく、挙動の一致は構造上の帰結となる。

---

## 6. 今後の拡張ポイント

- **読み(よみがな)検索**: 漢字を読みで参照する機能。辞書依存となるため、OS の読み付与
  機能(OSごとに辞書が異なり一致しない)は使用せず、形態素解析器と固定辞書をコアに同梱して
  両OS一致を保つ方針を採る。
- **セマンティック検索**: 埋め込みベクトルと近似最近傍探索による検索。同一コアに実装すれば、
  正規化および既存戦略と同様に一貫して管理できる。
- **二段ランキング**: 緩く畳み込んだ正規化で広く候補を取得し、より厳密な正準形に一致する
  ものを上位に配置する手法。再現率と適合率の両立を目的とし、可変正規化と戦略の組み合わせ
  という既存の枠組みの延長として位置づけられる。

# 関連知識の解説

設計判断の背景にある技術を、判断の根拠とともに解説する。

## A. Unicode 正規化(NFC / NFD / NFKC / NFKD)

同一の字形が複数のコードポイント列で表現されうるため、比較・検索の前に正規化して表現を
一意化する必要がある。正規化形は2軸の組み合わせにより4種類に分類される。

| | 合成(Composed) | 分解(Decomposed) |
|---|---|---|
| **正準等価**(字形・意味が同一) | NFC | NFD |
| **互換等価**(意味は対応するが体裁が異なる) | NFKC | NFKD |

- **正準(Canonical)**: `が`(単一文字)と `か`+濁点のような、同一の文字に対する表現の
  差異を統一する。
- **互換(Compatibility, K)**: 全角 `Ａ` と半角 `A`、半角カナ `ｶ` と全角 `カ`、丸数字 `①` と
  `1` のような、意味は対応するが体裁が異なる文字を統一する。全角半角の畳み込みはこの K が
  担う。
- **合成と分解**: 合成は結合済みの単一文字へ統一し、分解は基底文字と結合文字へ分離する。

### 基底処理に NFKC を採用する根拠

- 全角半角を畳み込むには互換(K)が必要である。NFC / NFD では半角カナが畳み込まれない。
- 濁点は既定で区別するため、合成(C)を採用する。分解形(NFKD)では `が` が `か`+結合濁点へ
  分離し、濁点を保持する目的に反してキーが不安定化する。NFKC では `が` は単一の合成文字
  として安定する。

> 補足: 濁点も曖昧化する要件の場合は、分解後に結合濁点・結合半濁点を除去する方法がある。
> 本エンジンは既定で濁点を区別するため、この除去は行わない(必要に応じて畳み込みの一軸として
> 追加できる)。

## B. 結合文字と濁点

- 結合濁点・結合半濁点は前の文字に重ねて表示される結合文字であり、単独表示用の記号とは
  別のコードポイントである。
- 半角の濁点・半濁点は、互換正規化により前の半角カナと合成され、全角の合成済み濁点付き
  カナへ統一される(例: 半角の `ｶ`+`ﾞ` → `が`)。これにより半角入力の濁点も単一文字に
  統一される。

## C. 全文検索と trigram(3文字分割)トークナイズ

- **トークナイザの選択が日本語処理の要点となる**。
  - 空白区切りを前提とするトークナイザは、分かち書きしない日本語ではほぼ機能しない。
  - **3文字分割(trigram)**: 文字を3文字単位に分割する。空白のない日本語でも部分一致が
    成立し、辞書を用いないためプラットフォーム非依存である。
  - 形態素解析(辞書依存): 単語境界で正確に分割でき精度は最も高いが、辞書の同梱が必要で
    処理が重く、辞書の差異が一致を脅かす。
- **3文字分割の制約**: 3文字未満のクエリは参照できないため、本設計では部分一致への
  フォールバックで補う。
- **関連度指標**: 語の頻度と文書長を加味した指標で並べ替える。

## D. 言語間バインディングの自動生成(UniFFI)

Rust で記述したロジックから、Swift / Kotlin 等の各言語向けバインディングを自動生成する
機構である。実装が単一であるため、クロスプラットフォームのアルゴリズム一致を構造的に
保証できる。値オブジェクト、メソッドを持つクラス、例外型、公開関数といった区別を保った
まま、各言語の慣用的な形式(Swift の struct・例外、Kotlin の data class・例外など)へ
対応づけられる。非同期処理も各言語の非同期機構へ対応づけられる。

実装を単一に保ち、両OSが同一のコードを介して動作することにより、正規化および検索の一致は
構造的に保証される。
